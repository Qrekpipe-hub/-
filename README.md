# Task Tracker — Recurrence Feature

Расширение модуля трекера задач периодичностью (тестовое задание).

---

## Быстрый старт

```bash
# Клонировать репозиторий
git clone <repo-url>
cd taskservice

# Запустить всё (PostgreSQL + миграции + API)
docker compose up --build

# API доступен на http://localhost:8080
```

Без Docker:
```bash
# Применить миграции (golang-migrate)
migrate -path ./migrations -database "$DATABASE_URL" up

# Запустить сервер
DATABASE_URL=postgres://... ADDR=:8080 go run ./cmd/api
```

## API

### Существующие эндпоинты (без изменений)

| Метод  | Путь                 | Описание               |
|--------|----------------------|------------------------|
| POST   | `/api/v1/tasks`      | Создать задачу         |
| GET    | `/api/v1/tasks`      | Список всех задач      |

| PUT    | `/api/v1/tasks/{id}` | Обновить задачу        |
| DELETE | `/api/v1/tasks/{id}` | Удалить задачу         |

| Метод | Путь                              | Описание                                              |

## Типы периодичности

Поле `recurrence` передаётся при создании/обновлении задачи. Примеры:

### 1. Ежедневно каждые N дней (`daily`)

```json
{
  "recurrence": {
    "type": "daily",
    "interval": 3
  }
}
```
Последовательность отсчитывается от `due_date` шаблонной задачи.  
`interval: 1` — каждый день, `interval: 2` — через день, и т.д.

### 2. Ежемесячно в определённое число (`monthly`)

```json
{
  "recurrence": {
    "type": "monthly",
    "day_of_month": 15
  }
}
```
Допустимые значения: 1–30.  
Если в месяце меньше дней чем `day_of_month` (например, `day_of_month: 30` в феврале) — **месяц пропускается**.

### 3. На конкретные даты (`specific_dates`)

```json
{
  "recurrence": {
    "type": "specific_dates",
    "dates": ["2024-06-01", "2024-07-04", "2024-12-25"]
  }
}
```

### 4. Чётные / нечётные дни месяца (`even_odd`)

```json
{
  "recurrence": {
    "type": "even_odd",
    "parity": "even"
  }
}
```
`parity`: `"even"` (2, 4, 6…) или `"odd"` (1, 3, 5…).

## Примеры запросов

### Создать периодическую задачу

```bash
curl -X POST http://localhost:8080/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Ежедневный обзвон пациентов",
    "description": "Связаться со всеми плановыми пациентами",
    "status": "pending",
    "due_date": "2024-06-01T00:00:00Z",
    "recurrence": {
      "type": "daily",
      "interval": 1
    }
  }'
```

### Предпросмотр дат (без создания)

```bash
curl "http://localhost:8080/api/v1/tasks/{id}/occurrences?from=2024-06-01&to=2024-06-30"
```

Ответ:
```json
{
  "task_id": "...",
  "dates": ["2024-06-01", "2024-06-02", "2024-06-03", ...]
}
```

### Сгенерировать экземпляры задачи

```bash
curl -X POST http://localhost:8080/api/v1/tasks/{id}/generate \
  -H "Content-Type: application/json" \
  -d '{"from": "2024-06-01", "to": "2024-06-30"}'
```

Создаёт отдельные задачи на каждую дату и возвращает их массивом. Каждый экземпляр имеет `parent_id`, указывающий на шаблон.

## Архитектурные решения и допущения

### Как хранится периодичность

`recurrence` хранится как `JSONB`-колонка в таблице `tasks`. Альтернатива — отдельная таблица `task_recurrences`, но это усложнило бы запросы без реальной выгоды: правило неотделимо от задачи и не является справочником.

### Шаблон vs. экземпляр

Введено разделение:
- **Шаблонная задача** — задача с заполненным `recurrence`. Создаётся пользователем как обычная задача + правило повторения. Сама по себе является задачей на `due_date`.
- **Экземпляр** — дочерняя задача, порождённая из шаблона. Имеет `parent_id`, не несёт правила повторения, статус всегда `pending` при создании.

Это позволяет каждому экземпляру жить самостоятельно: его можно завершить, отредактировать, удалить — независимо от остальных.

### Два эндпоинта вместо одного

`GET /occurrences` — «покажи мне, что будет создано» (dry-run). Врач может проверить даты перед тем как действительно их создавать.  
`POST /generate` — фактически создаёт записи в БД.

Это важно: генерация — изменяющая операция, а HTTP GET не должен иметь побочных эффектов.

### Ограничение диапазона в 2 года

Случайный запрос `generate` с диапазоном в 10 лет и `interval: 1` создаст ~3650 записей. Ограничение в 2 года — разумный барьер, достаточный для реального планирования. Значение можно вынести в конфиг.

### `day_of_month` ограничен 30, а не 31

День 31 существует лишь в 7 из 12 месяцев, что делает правило непредсказуемым. День 30 присутствует во всех месяцах кроме февраля, что ожидаемо. Это соответствует постановке задачи («от 1 до 30»).

### Что происходит с февралём при `day_of_month: 29/30`

Месяц **пропускается**. Альтернатива — перенос на последний день месяца — кажется менее предсказуемой: пользователь, ставящий задачу «30-го числа», не ожидает её 28 февраля.

### Повторный вызов `generate` на тот же диапазон

Создаст дубли. Идемпотентность не реализована намеренно, чтобы не усложнять MVP. В реальной системе стоит добавить `UNIQUE (parent_id, due_date)` и обработку конфликта через `ON CONFLICT DO NOTHING`.
